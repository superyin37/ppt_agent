## 建议书大纲框架（遵循 manus.md 40 页结构）

### A. 背景研究（约 11 页）

1. **政策解读 1** — 最相关的一项宏观政策（国家/省级）对本项目的直接影响
2. **政策解读 2** — 第二项关键政策（城市/行业专项）的落地机遇
3. **政策综合影响** — 两项政策叠加后对项目定位的综合研判
4. **上位规划** — 城市总规/控规/片区规划对地块的核心指引
5. **交通与可达性** — 地铁/公交/自驾可达性，与目标客群的匹配度
6. **文化与人文资源** — 周边历史文脉、文化IP、社区特征
7. **经济与消费数据 1** — 区域经济体量、商业密度、消费升级趋势
8. **经济与消费数据 2** — 目标客群消费力、客单价、到访频次预测
9. **经济综合结论** — 市场机会窗口与风险因素
10. **章节小结** — 背景研究的核心发现与项目机遇

### B. 场地分析（约 7 页）

11. **区位概况 1** — 城市坐标、行政区位
12. **区位概况 2** — 片区级区位、半径 3km 内主要节点
13. **区位概况 3** — 地块微区位、周边界面现状
14. **区位概况 4** — 用地红线、技术经济指标（容积率/建筑密度/绿化率）
15. **POI 分析** — 周边 500m 内商业/餐饮/教育/医疗分布
16. **场地综合** — 地块优势与制约条件矩阵

### C. 竞品分析（约 3 页）

17. **本地竞品** — 同城同类项目对比（定位/体量/运营/口碑）
18. **网络竞品** — 同类型标杆项目（全国/国际）品质标杆
19. **竞品结论** — 差异化定位策略的空白机会

### D. 参考案例（2-5 页，按选择数量）

20+. **参考案例 N** — 每个案例一页：核心亮点 + 对本项目的启发

### E. 项目定位

- **项目战略定位** — 基于背景研究 + 竞品分析形成的定位主张（一句话）

### F. 设计策略

- **设计策略总纲** — 从定位主张演绎出的 3-5 条设计原则
- **概念方案介绍** — 核心设计概念（功能/空间/形态）

## 输出格式要求

返回一个 JSON 对象，结构如下：

```json
{
  "brief_title": "项目全称 设计建议书",
  "executive_summary": "200字以内的项目核心判断",
  "chapters": [
    {
      "chapter_id": "background",
      "title": "背景研究",
      "key_findings": ["发现1", "发现2", "发现3"],
      "narrative_direction": "本章向甲方传递的核心信息"
    },
    ...
  ],
  "positioning_statement": "基于研究的项目定位主张（1-2句话）",
  "design_principles": ["原则1", "原则2", "原则3"],
  "recommended_emphasis": {
    "policy_focus": "最核心的政策机遇",
    "site_advantage": "最大场地优势",
    "competitive_edge": "相对竞品的核心差异化",
    "case_inspiration": "参考案例给予的最重要启发"
  },
  "narrative_arc": "整体叙事逻辑（从背景到定位到设计的逻辑链）"
}
```

## 规则（严格控制输出长度）

- `executive_summary`：≤100 字，必须包含城市、建筑类型、规模等实际数据
- `chapters`：最多 5 个章节，每章 `key_findings` ≤ 3 条，每条 ≤ 30 字；`narrative_direction` ≤ 30 字
- `positioning_statement`：≤50 字，必须可量化或可验证，避免「高端」「国际化」等空话
- `design_principles`：最多 4 条，每条 ≤ 30 字，以动词开头
- `recommended_emphasis` 每项 ≤ 30 字
- `narrative_arc`：≤80 字，给 Outline Agent 的叙事指令
- 充分利用 `<available_data>` 中的具体数据，而非生成假设性内容
- **输出必须是完整合法的 JSON，宁可内容简短，不可截断**
