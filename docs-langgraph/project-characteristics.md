# 项目特征说明

## 总体定位

PPT-Maker 当前是一个“固定结构的建筑方案汇报生成器”。它读取 `data/case_<id>/` 下的素材包，把输入资料解析成结构化数据，再生成一份固定 40 页的 HTML deck。

它不是一个自由对话式 PPT agent，也不是“给一个主题就从零生成任意页数 PPT”的系统。当前项目的页面数量、页面顺序、章节结构、组件类型和视觉模板都有明确的固定规则。

## 当前固定生成 40 页

当前项目固定生成 40 页。这个页数不是运行时由模型决定的，也不是根据素材包自动增减。

如果某个节点没有产出某一页，`aggregate_specs` 会补一个 `[missing page N]` 占位页，保证最终仍然是 40 页。

因此，生成结果可能有内容缺失或占位页，但正常情况下页码范围仍然是 1 到 40。

## 页面顺序是固定的

40 页的主题和位置基本固定。例如：

第 1 页是封面，第 2 页是目录，第 3 页是项目背景转场页，第 4-7 页是政策相关内容，第 23-25 页是参考案例，第 29-37 页是 3 个概念方案的 9 个视角，第 40 页是结束页。

不同素材包会改变这些页里的文字、图片、表格或图表，但不会自动改变“第几页讲什么”的大框架。

## 当前 pipeline 不调用豆包 LLM 做文本生成

当前 pipeline 没有任何节点调用豆包 LLM 做文本生成。

`DoubaoClient` 已经存在，提供 `chat()` 和 `structured()` 能力，但目前没有被任何节点引用。它是后续扩展预留，不是当前生成链路的一部分。

因此，当前 deck 中的文字不是豆包现场写出来的。文字主要来自输入资料解析、Python 硬编码文案、规则拼接、Excel 表格映射、Tavily 搜索结果和少量启发式计算。

## `parse_outline` 是规则解析，不是 LLM 抽取

设计大纲由 `parse_outline` 节点解析。这个节点使用 markdown 标题拆分、编号项提取、markdown 表格解析和字段名匹配。

它不会把大纲发给豆包，也不会让模型判断哪些内容属于政策、区位、文化或方案。

这意味着输入大纲的格式很重要。标题、编号、字段名或表格结构偏离预期时，解析结果可能为空或不完整。

## 当前没有 LLM 审查

当前 pipeline 没有 LLM 审查节点。

系统不会让模型复核政策摘要是否准确，不会判断项目定位是否合理，也不会自动检查图文是否匹配。

如果输入资料中有错误，或者规则解析漏掉了字段，当前系统通常不会主动发现内容质量问题。

## 当前只做完整性检查

`validate` 节点只做完整性检查，不做内容质量检查。

它主要检查三件事：

1. 是否正好有 40 页。
2. 页码是否连续。
3. `index.html` 是否存在。

它不会检查某一页是否写得好，不会检查表格数据是否真实，不会检查图表是否表达正确，也不会检查视觉上是否重叠。

## 当前没有截图回看

项目当前不会在 HTML 渲染后自动截图，也不会用视觉模型回看页面。

也就是说，系统不会自动发现某页文字溢出、图片比例不佳、内容遮挡、颜色不协调或页面空白。

这些问题需要通过人工打开 `index.html`、使用 `inspect` 查看 `SlideSpec`，或后续新增审查节点来处理。

## 视觉设计由固定 Jinja2 模板决定

每一页的背景颜色、基础排版、页眉页脚、章节标签、页码样式、组件布局和大部分视觉规则，都由固定的 Jinja2 模板决定。

内容节点只负责产出 `SlideSpec`，例如这一页用什么组件、标题是什么、`data` 里有什么字段。真正把这些字段排成页面的是模板系统。

因此，节点通常不会决定“这页背景是什么颜色”“标题放左边还是右边”“图片占多宽”“页码怎么显示”。

## 同一个组件的视觉结构是稳定的

如果两份素材包都在第 4 页使用 `policy_list` 组件，那么第 4 页的卡片结构、标题层级、行距、字体、背景和页脚风格应该保持一致。

不同的是政策标题、政策内容、影响描述、来源链接这些数据。

换句话说，组件决定视觉骨架，素材包决定填进去的内容。

## 不同素材包生成的 PPT 会有相同的视觉规则

只要使用同一个模板，例如 `minimalist_architecture`，不同素材包生成的 PPT 会共享同一套视觉规则。

同一固定页的排版、背景色、章节色、页码位置、组件布局和字体系统应该相同。

差异主要来自：

- 标题文字不同。
- 正文内容不同。
- 表格行列不同。
- 图表数据不同。
- 输入图片不同。
- RunningHub 生成图不同。
- Tavily 搜索结果不同。

## 模板不会根据素材包自动重新设计

当前模板不会根据项目风格自动改变整体视觉语言。

例如，素材包里写“极简主义”“新中式”或“科技风”，这可能会影响某些文案或图像 prompt，但不会让 Jinja2 模板自动变成另一套版式系统。

如果要换视觉风格，应新增或修改模板，而不是期待 pipeline 自动设计新风格。

## 章节颜色由模板固定映射

页面属于哪个章节，是通过页码范围判断的。章节对应的中文、英文标签和颜色由模板和 renderer 固定。

例如 1-12 页属于项目背景，13-19 页属于场地分析，20-26 页属于项目定位，27-40 页属于设计方案。

如果总页数或章节划分变化，需要同步修改 renderer、模板和文档中的页码表。

## 图片会变化，但图片槽位通常不变

不同素材包的图片内容会变，但图片出现的位置通常由页面组件固定。

例如 `chart` 组件会把图表放在模板规定的位置，`image_grid` 会按固定网格渲染图片，`concept_scheme` 会按固定方案页布局展示概念图。

素材包决定图片是什么，模板决定图片放在哪里、占多大、周围有哪些文字。

## 图表是代码生成的，不是模板预画的

政策影响图、POI 分类图等图表由 Python 代码和 matplotlib 生成。

模板只是把生成好的图表路径读出来，再通过 `embed_image()` 内联进 HTML。

因此，图表里的数值、标签和柱形高度可能随素材变化；图表所在页面的布局和视觉容器由模板决定。

## RunningHub 用于图像生成，不用于文字生成

RunningHub 当前用于生成文化图、概念方案图、logo 或目录插图。

它是图像生成服务，不是豆包文本 LLM。它不会负责撰写政策解读、项目定位、设计任务书或表格内容。

缺少 RunningHub key、使用 dry-run 或调用失败时，项目会写 SVG 占位图，deck 仍然可以生成。

## Tavily 只用于联网检索页

Tavily 当前主要用于第 22 页同类产品检索。

未配置 `TAVILY_API_KEY` 时，项目不会中断，而是生成未配置说明或空结果。

Tavily 返回的是搜索结果，不是 LLM 改写后的分析报告。

## 有些页面是硬编码文案

当前项目中有些页面主要来自 Python 硬编码文案。

例如部分章节转场页、结束页、综合总结页包含固定表达或固定结构。它们可能会读取少量输入字段，但不是由 LLM 根据全量资料重新总结。

如果这些页面看起来“不像当前项目”，通常应该直接改对应节点里的字面量，或者新增更智能的节点覆盖它们。

## 有些页面强依赖输入格式

政策页、产业政策页、上位规划页、区位页、案例页和方案 seed 都依赖输入 markdown 的标题、编号和字段格式。

POI 页依赖 Excel sheet 名称和列名映射。

输入文件存在但格式不符合预期时，系统可能无法解析出内容，最后表现为空表、少内容或缺页占位。

## 当前不做语义检索或 RAG

项目当前没有向量数据库，没有 RAG，也不会把所有输入资料切片后做语义检索。

节点读取的是明确的文件、明确的语义键和明确的字段。它更像一个规则化资料装配 pipeline，而不是开放式知识问答系统。

## `SlideSpec.data` 不做严格字段校验

`SlideSpec` 会校验页码、组件名、标题等外层字段，但 `data` 内部是松耦合字典。

这意味着模板需要的字段如果写错、少写或层级不一致，系统通常不会报错，只会在页面上显示为空。

这是当前模板调试中最重要的特征之一。

## 缺页会补占位，不代表内容正确

最终生成 40 页并不代表每一页内容都正确。

如果某页是 `[missing page N]`，说明它是兜底补出来的，不是正常内容节点生成的。

即使没有缺页，也仍然可能存在字段为空、图片占位、表格数据少、内容质量不足等问题。

## checkpoint 会影响调试结果

同一个 case 会使用固定 `thread_id` 和 `checkpoint.sqlite`。

这可以避免重复跑昂贵的图像生成，但也可能导致改了代码后仍然看到旧结果。

调试节点逻辑时，应使用 `--force` 删除 checkpoint；只改模板时，应使用 `render-only`。

## dry-run 会跳过外部图像生成

`--dry-run` 和 `--no-images` 当前都会让 `state["dry_run"] = True`。

dry-run 下图像相关节点直接走占位路径，不调用 RunningHub。它适合调试流程、模板和数据结构，不适合评估最终图像质量。

## 单文件 HTML 是有意选择

最终输出是单文件 `index.html`，图片会被 base64 内联。

好处是容易打开、移动和分享。代价是文件可能很大，真实图像较多时浏览器打开会更慢。

当前项目没有把图片拆成旁边的 assets 目录再由 HTML 引用，也没有上传 CDN。

## 当前输出是 HTML deck，不是 `.pptx`

项目生成的是 HTML 演示文稿，不是 PowerPoint 原生 `.pptx` 文件。

HTML deck 可以在浏览器中打开，也方便用前端模板控制视觉，但它不是 Office PowerPoint 的文件格式。

如果需要 `.pptx`，需要额外增加导出链路或转换工具。

## 项目更适合“固定汇报框架下的批量生成”

当前架构适合在固定 40 页框架下，为不同素材包批量生成同结构、同视觉系统的建筑方案汇报。

它不适合直接承担“任意主题、任意页数、自动设计版式、自动审查内容”的完整 PPT agent 角色。

如果要向那个方向扩展，需要新增 LLM 文本节点、内容审查节点、视觉回看节点、可配置页码结构和更严格的数据契约。
