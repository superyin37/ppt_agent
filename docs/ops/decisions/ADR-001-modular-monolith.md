# ADR-001:采用模块化单体而非微服务

## Context

项目启动初期,业务逻辑(多 Agent 协作、异步任务、向量检索、HTML 渲染)尚未稳定,需要快速迭代验证产品形态。

## Options Considered

| 方案 | 优点 | 缺点 |
|------|------|------|
| 微服务架构 | 独立扩展、团队解耦、技术栈自由 | 运维复杂、跨服务调试困难、早期开发成本高 |
| **模块化单体**(选) | 开发快、调试容易、部署简单 | 水平扩展有限、耦合风险 |

## Decision

**选模块化单体**。通过目录分层(`agent/` / `tool/` / `api/` / `render/` / `schema/` / `tasks/`)保持逻辑分离,未来拆分时边界清晰。

异步工作负载通过 Celery + Redis 隔离,相当于"逻辑上的服务拆分"而无真实进程边界。

## Consequences

### 好处
- 从 0 到端到端跑通只用了约 2 个月
- 调试方便:一个 IDE 能看全链路
- 部署简单:一个 docker-compose 搞定

### 代价
- 单体进程扩展受限,高并发场景需提前规划
- 模块边界靠约定,没有进程隔离强制

### 重新评估条件
- 如果日活用户 > 1000,考虑拆分 render / compose 为独立服务
- 如果团队规模 > 5 人并行开发,考虑拆 API 与 Worker
