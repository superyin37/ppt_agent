# 阶段八：审查与修复

> 可选循环阶段。对渲染后的幻灯片进行多层审查，发现问题后重新编排+渲染，最多迭代 N 次。

## 触发

```
POST /projects/{project_id}/review
→ api/routers/render.py (line 50)
→ Celery 任务: review_slides (tasks/review_tasks.py)
```

## 审查层级

| 层 | 名称 | 检查内容 | 工具 |
|----|------|---------|------|
| `rule` | 布局规则 | 文字溢出、图片宽高比、间距合规 | 纯逻辑检查 |
| `semantic` | 内容语义 | 信息清晰度、视觉层次、一致性 | LLM（可选） |

规则详见 `docs/11_review_rules.md`。

## 修复循环

```
对每页 slide（status == "rendered"）：
  ① 执行规则审查
  ├── 通过 → slide.status = "review_passed"
  └── 失败（P0/P1 级问题）→
        slide.status = "repair_needed"
        → compose_slide(entry, theme, assets, binding)  // 重新编排
        → render_slide_html(slide, theme, assets)       // 重新渲染
        → screenshot_slide(html)                        // 重新截图
        → 再次审查（最多 3 次迭代）
          └── 仍失败 → slide.status = "failed"
              或 → escalate_human
```

## 审查严重级别

| 级别 | 含义 | 处理 |
|------|------|------|
| `P0` | 阻塞性问题（内容溢出/不可读） | 强制修复 |
| `P1` | 重要问题（布局不合理） | 尽量修复 |
| `P2` | 优化建议 | 记录但不修复 |
| `PASS` | 通过 | 继续 |

## SlideStatus 转换

```
rendered
  → review_pending
  → review_passed     (P2/PASS)
  → repair_needed     (P0/P1)
    → repair_in_progress
    → rendered          (重新渲染后)
    → review_passed     (修复成功)
    → failed            (超过最大迭代或 P0 无法修复)
```

## 产出

`Slide.review_result_json` 示例：

```json
{
  "decision": "repair_required",
  "severity": "P1",
  "issues": [
    {
      "type": "text_overflow",
      "region": "right",
      "description": "right 区域文字超出边界，约 3 行溢出",
      "suggestion": "减少 bullet-list 条目或缩小字号"
    }
  ],
  "iteration": 1
}
```
