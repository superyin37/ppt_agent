# 阶段四：素材绑定

> **无 LLM 调用**，纯正则匹配。将大纲每页的 `required_input_keys` 与实际 MaterialItem/Asset 进行匹配，产出每页的绑定清单。

## 触发

```
POST /projects/{project_id}/outline/confirm
→ api/routers/outlines.py (line 189)
→ _compose_render_worker()
  → bind_outline_slides(project_id, outline_id, db)  ← 本阶段
  → compose_all_slides(...)
```

## 执行流程

```
1. 加载 Outline（含 spec_json → OutlineSpec）
2. 加载最新 MaterialPackage
3. 加载所有 MaterialItem（package 下）
4. 加载所有派生 Asset（package 下）
5. for entry in outline_spec.slides:
     binding = _build_binding(entry, items, assets)
     db.add(binding)
6. db.commit()
```

## `_build_binding()` 内部逻辑

```
a. _collect_required_patterns(entry)
   ├── entry.required_input_keys 非空 → 直接使用
   └── 否则 → 从 PPT_BLUEPRINT 找对应 slot.required_inputs

b. expand_requirement(key) → patterns
   （一个 key 可能展开为多个正则模式）

c. find_matching_items(patterns, items)
   → logical_key.startswith(pattern.rstrip("*"))

d. find_matching_assets(patterns, assets)
   → 相同逻辑

e. coverage_score = (required - missing) / required

f. evidence_snippets = [item.text_content[:200] for matched_items]
```

## 产出

每页一条 [[schemas/SlideMaterialBinding]]，关键字段：

| 字段 | 用途 |
|------|------|
| `derived_asset_ids` | 传给 ComposerAgent 的可用资产范围 |
| `evidence_snippets` | 传给 ComposerAgent 的文本证据 |
| `coverage_score` | 质量监控 |
| `missing_requirements` | 标记素材缺口 |
